**Задача:** разработать дашборд для сервисного центра, который поможет анализировать обращения клиентов, выявлять самые частые поломки, оценивать загрузку мастеров и находить проблемные бренды/модели.

**Результат:** трёхстраничный дашборд в Power BI с ключевыми метриками, анализом проблем и клиентскими отчётами.

**Инструменты:** Power BI Desktop, DAX, Power Query.

---

## Предобработка данных

Исходные данные — три выгрузки из учётной системы (2024, 2025, 2026).  
В Power Query выполнены следующие шаги:

### 1. Объединение источников
- Три таблицы объединены в одну через `Table.Combine`
- Удалены дубликаты и служебные столбцы

### 2. Приведение типов и очистка
- Пропуски в поле `Сумма ремонта` заменены на `null`
- Удалены строки с нулевой суммой
- Ошибки в датах обработаны через `try ... otherwise`

### 3. Нормализация названий техники
- Единый формат: `К/м Бренд Модель`
- Исправлены опечатки (Delongjhi → Delonghi, Phillips → Philips)
- Унифицированы сокращения (К/в → К/м)

### 4. Извлечение бренда и модели
- Бренд — второе слово из `Наименование техники`
- Модель — третье слово
- Дополнительная чистка через замены и маски (`EA82*` → `EA82`)

### 5. Классификация проблем
- На основе поля `Неисправность со слов клиента` создана категория `Тип неисправности`
- Правила: "протек"/"течь" → Протечка, "молк"/"помол" → Кофемолка, "диагностик" → Диагностика и т.д.

### 6. Обогащение через мердж
- Таблица заказов соединена со справочником `User_maps` для получения `user_id`
- Тип соединения: левое внешнее (Left Outer)

### 7. Создание новых полей

| Поле | Описание |
|------|----------|
| `Длительность ремонта (дни)` | Разница между `Дата отдачи` и `Дата приёма` |
| `Группа чека` | Категории: до 3 тыс, 3-7 тыс, 7-15 тыс, более 15 тыс |
| `Группа по обращениям` | 1 обращение / 2 обращения / 3+ обращений |
| `Бренд_модель` | Объединённое поле для фильтрации |

### 8. Создание таблицы `Календарь`
Отдельная таблица дат для корректной работы временных фильтров и иерархий. Связана с основной таблицей по полю `Дата заказа`.

### 9. Модель данных
- Связь "один ко многим" между `Календарь` и `Заказы`
- Отсутствие циклических зависимостей
- Все меры вынесены в отдельную папку "Измерения"

### 10. Обезличивание данных
Данные анонимизированы для публичного портфолио:
- Имена клиентов заменены на `User_id`
- Точные адреса удалены
- Суммы изменены

### 11. Итоговая таблица
Отобраны столбцы: `Дата`, `User_id`, `Бренд`, `Модель`, `Тип неисправности`, `Сумма ремонта`, `Дата отдачи`
